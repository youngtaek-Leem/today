# CLAUDE.md — 소프트웨어 개발 표준 워크플로우

> Claude Code가 이 프로젝트에서 소프트웨어를 개발할 때 따르는 규칙입니다.
> 범용 템플릿입니다. 새 프로젝트를 시작할 때 이 파일을 프로젝트 루트에 복사해서 사용합니다.

## 핵심 원칙

1. **Requirement.md가 단일 기준 문서(Single Source of Truth)다.** 모든 개발은 이 문서를 따른다.
2. **계획 없이 코드를 작성하지 않는다.** 사양이 정해지면 먼저 계획을 세우고 사용자 승인을 받는다.
3. **모든 모듈은 독립적으로 검증 가능해야 한다.** 검증할 수 없는 단위는 모듈이 아니다.
4. **검증을 통과하지 않은 코드는 완성이 아니다.** "작성했다"가 아니라 "검증했다"가 완료의 기준이다.

- 주요 명령:
  - 설치: []
  - 실행: []

> 이 섹션이 비어 있으면 프로젝트 초기 설정 시 실제 값으로 채우고, 구조나 명령이 바뀌면 즉시 갱신한다.

## 개발 워크플로우

### 1단계 — 사양 협의 및 계획 수립

- 사용자가 개발 사양을 제시하면 곧바로 코드를 쓰지 않고 **먼저 개발 계획 초안을 작성해 제시한다.**
- 계획 초안에 포함할 항목:
  - 목표와 범위 (포함하는 것 / 제외하는 것)
  - 기술 스택과 그 선택 이유
  - 모듈 분해안 (각 모듈의 책임과 검증 방법 포함)
  - 개발 순서 (의존성 기준)
  - 예상 리스크와 미결정 사항
- 사용자와 피드백을 주고받으며 계획을 다듬는다. 미결정 사항은 선택지와 추천안을 붙여 질문하고, 한 번에 핵심 질문 위주로 묶어서 묻는다.
- **사용자가 계획을 승인하면 Requirement.md에 저장한다.** 승인 전에는 본 개발을 시작하지 않는다.

### 2단계 — 모듈 분해

- 승인된 계획을 검증 가능한 모듈 단위로 분리한다. 각 모듈은 다음을 반드시 정의한다:
  - 책임 (이 모듈이 하는 일 한 문장)
  - 입출력 인터페이스 (다른 모듈과 주고받는 것)
  - 의존 모듈
  - 검증 방법 (단위 테스트, 실행 스크립트, 수동 확인 절차 중 구체적으로 명시)
  - 완료 기준
- 모듈 간 의존성을 최소화하고, 의존성이 적은 모듈부터 개발 순서를 정한다.
- 분해 결과를 Requirement.md의 모듈 표에 기록한다.

### 3단계 — 모듈 개발 루프 (반복)

각 모듈에 대해 다음 사이클을 반복한다:

1. **구현**: Requirement.md에 정의된 인터페이스와 완료 기준에 맞춰 작성한다.
2. **검증**: 해당 모듈에 정의된 검증 방법을 실제로 실행한다.
3. **개선**: 실패하면 원인을 분석하고 수정한 뒤 재검증한다. 통과할 때까지 반복한다.
4. **기록**: 통과하면 Requirement.md의 모듈 상태를 갱신하고 다음 모듈로 진행한다.

- 새 모듈이 완성되면 이미 완성된 모듈의 검증도 다시 실행해 회귀(regression)가 없는지 확인한다.
- 검증 중 사양의 모호함이나 설계 결함이 드러나면 조용히 우회하지 말고 보고하고, 합의된 내용을 Requirement.md에 반영한다.

### 4단계 — 통합 및 완료

- 모든 모듈 완성 후 통합 테스트를 수행한다.
- Requirement.md의 요구사항 목록과 실제 구현을 하나씩 대조한다.
- 최종 보고: 완성된 것, 검증 결과, 미완성이거나 보류된 것을 숨김없이 정리한다.

## Requirement.md 관리

- 위치는 프로젝트 루트. 없으면 1단계 절차를 거쳐 생성하고, 있으면 **개발 작업 시작 전에 반드시 읽는다.**
- 다음 경우 즉시 업데이트하여 항상 최신 상태를 유지한다:
  - 사용자가 새 기능을 요청했을 때
  - 개발 중 모호한 부분이 발견되어 사양을 확정했을 때
  - 설계나 모듈 구조가 변경되었을 때
  - 모듈의 상태(개발 중 → 검증 통과 등)가 바뀌었을 때
- 문서와 코드가 다르면 코드가 아니라 문서를 기준으로 삼되, 어느 쪽이 맞는지 사용자에게 확인한다.
- 권장 구조:
  1. 개요 및 목표
  2. 기능 요구사항 (번호 부여, 추적 가능하게)
  3. 기술 스택 및 제약 조건
  4. 모듈 분해표 — `모듈 | 책임 | 인터페이스 | 의존성 | 검증 방법 | 상태`
  5. 미결정·보류 사항
  6. 변경 이력 (날짜, 변경 내용, 사유)

## 승인 정책

- **프로젝트 폴더 안에서 이루어지는 작업은 별도 확인 없이 바로 진행한다:**
  - 파일·폴더의 생성, 수정, 삭제, 이동
  - 로컬 빌드, 테스트, 실행, 조회
  - 프로젝트 로컬 개발 의존성 설치 및 사용 (npm install, pip install, 가상환경 구성 등)
  - 개발에 필요한 도구의 프로젝트 범위 설치 및 사용
- **다음은 반드시 사용자 확인 후 진행한다:**
  - 프로젝트 폴더 밖의 파일 변경, 시스템 전역 설정 변경
  - git push --force, 원격 브랜치 삭제
  - DB 마이그레이션·시드 실행, 배포
  - 외부 API로의 실제 발송·결제
  - .env 등 시크릿 파일 수정
  - 복구 불가능한 대량 삭제

## 코드 작성 규칙

- 수정 전에 반드시 대상 파일을 실제로 읽는다. 읽지 않은 파일을 수정하지 않는다.
- 요청된 범위만 고친다. 요청 없는 리팩터링, 기능 추가, 스타일 변경은 하지 않는다. 도움이 될 것 같으면 작업 완료 후 제안만 한다.
- 존재하지 않는 API·함수·라이브러리를 추측으로 쓰지 않는다. 실제 코드, 타입 정의, 공식 문서에서 확인한 뒤 사용한다. 확인하지 못했으면 그 사실을 밝힌다.
- 기존 코드 스타일과 프로젝트 컨벤션을 따른다. 내 취향으로 바꾸지 않는다.
- 일관된 폴더 구조를 유지한다. 새 파일은 기존 구조의 논리에 맞는 위치에 만든다.
- **측정 없이 최적화하지 않는다.** 성능 문제는 프로파일링이나 측정으로 병목을 확인한 뒤에만 고친다. 추측에 의한 사전 최적화는 하지 않는다.
- 새 라이브러리 도입 전에 표준 라이브러리나 기존 의존성으로 충분한지 먼저 검토한다. 도입하면 버전을 고정한다(lockfile, requirements.txt 등).

## 검증 규칙

- 프로젝트 초기에 **린트·타입 체크·전체 테스트를 한 번에 실행하는 단일 검증 명령**(예: `make check`, `npm run check`)을 만들고, 모듈 완성 시마다 이 명령으로 검증한다. 이것이 회귀 확인의 기본 수단이다.
- 모듈·작업의 완료 선언은 정의된 검증 방법을 실제로 실행해 통과한 뒤에만 한다.
- 테스트는 정상 경로뿐 아니라 경계값과 오류 경로도 함께 작성한다.
- 버그를 수정할 때는 그 버그를 재현하는 테스트를 먼저 추가하고, 수정 후 통과하는지 확인한다.
- 테스트가 없는 코드는 최소한 실행 확인이나 문법·타입 체크라도 수행한다.
- 명령이 실패하면 원인을 확인하고 방법을 바꿔 1~2회 재시도한다. 그래도 실패하면 실패 사실·원인·시도한 방법·대안을 보고한다. 성공한 척하지 않는다.
- 검증 결과(테스트 통과 수, 실패 내용)는 각색하지 않고 그대로 보고한다.

## 버전 관리

- 프로젝트 시작 시 git 저장소가 없으면 `git init`으로 생성한다.
- **모듈이 검증을 통과한 시점마다 커밋한다.** 문제가 생겼을 때 되돌릴 지점을 확보하는 것이 목적이다.
- 커밋 메시지는 변경 내용을 한 줄로 요약한다. 원격 반영(push)은 승인 정책을 따른다.

## 불확실성 처리

- 확신 없는 정보(라이브러리 동작, 버전별 차이, 설정의 효과)는 사실처럼 쓰지 않는다. 코드나 문서로 확인하거나 "확실하지 않다"고 밝힌다.
- 모호한 요청은 가장 그럴듯한 해석을 명시하고 진행한다. 해석이 크게 갈릴 때만 질문한다.
- 사용자가 오류를 지적하면 먼저 사실을 확인한다. 맞으면 간결히 인정하고 수정하고, 틀린 지적이면 근거를 들어 정중히 기존 판단을 유지한다.
- 내 이전 작업의 오류를 발견하면 지적받기 전에 먼저 정정한다.

## 보안

- 악성 코드(멀웨어, 익스플로잇, 랜섬웨어, 피싱, 계정 탈취, 서비스 마비 도구)는 어떤 명분으로도 작성·개선하지 않는다.
- 방어적 보안은 지원한다: 취약점 지적과 수정, 보안 설정 검토, 로그 분석.
- 시크릿(API 키, 비밀번호, 토큰)을 코드에 하드코딩하지 않는다. 기존 코드에서 발견하면 경고하고 환경변수 분리를 제안한다.
- 시크릿 값을 로그, 커밋 메시지, 출력에 노출하지 않는다.

## 완료 전 체크리스트

1. Requirement.md를 읽고 그 내용에 맞게 작업했는가?
2. 변경된 사양·설계·모듈 상태를 Requirement.md에 반영했는가?
3. 수정 대상 파일을 실제로 읽었는가?
4. 요청 범위를 벗어난 변경이 없는가?
5. 추측으로 쓴 API·함수가 없는가?
6. 모듈별 검증 방법을 실제로 실행해 통과했는가? 기존 모듈의 회귀 확인도 했는가?
7. 파급 효과가 큰 명령(폴더 밖 작업, 배포, 시크릿 등)에 사용자 확인을 받았는가?
8. 실패하거나 미완성인 부분을 숨기지 않고 보고했는가?
